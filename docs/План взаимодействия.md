### **План взаимодействия**

#### **1. Фаза 0: Подготовка и Синхронизация (1-2 дня)**

*   **Репозиторий:** Один человек создает Git-репозиторий на GitHub/GitLab. Добавляет второго как collaborator.
*   **Базовая структура:** Совместно в `main` ветке создается структура проекта:
    *   `CMakeLists.txt` (базовый, с подключением Qt).
    *   Папки: `src`, `include`, `data` (для словарей, JSON).
    *   `.gitignore` для C++/Qt/CMake.
*   **Определение API (Контракт):**
    *   **Ключевой шаг.** Создать один общий заголовочный файл (например, `core_api.h`), доступный для обоих модулей.
    *   В этом файле описать интерфейс Backend'а (класс `CoreEngine`) и все структуры данных для обмена (`SessionState`, `SessionMetrics`, etc.).
    *   Этот файл — **единственная точка соприкосновения** кода Frontend и Backend. Согласовать его до начала основной работы.

#### **2. Фаза 1: Параллельная разработка (Основной этап)**

*   **Система задач:** Использовать GitHub Issues для постановки задач. Одна фича = один Issue.
*   **Git Workflow (обязательно):**
    1.  Никто не коммитит напрямую в `main`.
    2.  Для каждой задачи (Issue) создается отдельная ветка (например, `feature/session-logic` или `feature/main-window`).
    3.  Работа ведется только в своей ветке.
    4.  Когда задача готова, создается Pull Request (PR) в `main`.
*   **Разделение ответственности:**
    *   **Исполнитель 1 (Backend):** Работает над логикой. Может создать простой консольный "драйвер" для тестирования своих модулей без UI. Его код не должен включать `#include <Q*>` за пределами API-структур, если это возможно.
    *   **Исполнитель 2 (Frontend):** Работает над UI. На начальном этапе использует "заглушки" (mock-объекты) вместо реального Backend, которые возвращают заранее заданные данные (`SessionState` с тестовым текстом), чтобы не ждать готовности логики.
*   **Коммуникация:**
    *   **Асинхронно:** Обсуждение конкретной задачи — в комментариях к соответствующему Issue/PR.
    *   **Синхронно:** Быстрые вопросы и общая координация — в мессенджере (Telegram/Discord).

#### **3. Фаза 2: Интеграция и Ревью (Регулярный процесс)**

*   **Code Review:**
    *   Перед слиянием Pull Request'а второй исполнитель **обязан** просмотреть код.
    *   **Цели ревью:** поиск ошибок, проверка соответствия общему стилю, понятность кода, соответствие API-контракту.
*   **Интеграционные сессии:**
    *   Раз в 2-3 дня (или по завершении крупной фичи) проводить интеграцию.
    *   Оба исполнителя созваниваются, один сливает свой PR в `main`, затем второй обновляет свою ветку из `main` (`git pull origin main`) и сливает свой PR.
    *   Совместно компилируют проект и проверяют, что Frontend корректно работает с реальным Backend.
*   **Обновление API:** Если в ходе работы нужно изменить API (добавить метод, изменить структуру), это выносится на общее обсуждение. После согласования оба исполнителя обновляют код в своих ветках. `main` всегда должен содержать стабильную, рабочую версию проекта.
